# Pinball_Logue 작업·변경·인수인계 규약

## 목적

대화 기록이나 외부 페이지 없이도 Git에 포함된 로컬 Markdown과 코드만으로 현재 기획, 변경 이유, 작업 상태와 다음 행동을 이해할 수 있게 한다.

## 로컬 기준과 권한

- 실행 기준은 저장소에 기록된 로컬 문서다.
- 작업자 또는 에이전트의 요청은 제안이며, 사용자가 승인하기 전에는 확정 기획이 아니다.
- 사용자가 현재 요청에서 외부 소스를 제공하고 읽기·갱신을 명시한 경우에만 해당 소스를 확인한다.
- 외부 소스에서 승인된 변경을 로컬 문서에 반영한 뒤에는 로컬 문서를 기준으로 작업한다.
- 로컬 정보가 부족하거나 충돌하면 외부 검색이나 추측 대신 사용자 피드백을 요청한다.

## 기능별 문서 구조

```text
<feature>/
├── AGENTS.md
├── SPEC.md
├── DECISIONS.md
└── .work/
    └── <JIRA-KEY 또는 task-slug>/
        ├── PLAN.md
        ├── QA.md
        ├── HANDOFF.md
        ├── daily/
        │   └── YYYY-MM-DD.md
        └── evidence/
```

모든 파일을 미리 만들지 않는다.

새 작업 문서는 `.agents/templates/`의 해당 템플릿을 대상 기능에 복사해 시작한다. 작업 조정자·구현자·전문 QA 역할과 프로젝트 스킬은 `.agents/AGENTS.md`를 따른다.

| 문서 | 생성 시점 | 책임 |
| --- | --- | --- |
| `AGENTS.md` | 기능 디렉토리를 만들 때 | 기능 책임, 제외 범위, 의존성, 읽을 문서와 검증 규칙 |
| `SPEC.md` | 첫 세부 기획을 승인할 때 | 현재 승인된 최신 기획과 수용 기준 |
| `DECISIONS.md` | 첫 중요 기획 변경을 승인할 때 | 변경 전후, 이유, 영향과 승인 상태 |
| `PLAN.md` | 실제 작업을 시작할 때 | 단계, 검증 계획, 승인 범위와 진행 상태 |
| `QA.md` | 최종 QA를 계획할 때 | QA 인수 목록, 환경, 결과와 증거 |
| `HANDOFF.md` | 작업자를 교체하거나 PR로 전달할 때 | 현재 상태, 남은 작업과 다음 첫 행동 |
| `daily/YYYY-MM-DD.md` | 구현 또는 검증한 날 | 구현 내용·결과·트러블슈팅·사용자 피드백 개선 기록 |

- 단일 기능 작업은 해당 기능의 `.work/`에 기록한다.
- 여러 기능에 걸친 작업은 주 책임 기능 하나를 정하고 그곳에 기록한다.
- 저장소 전체 정책 작업은 루트 `.work/`를 사용할 수 있다.
- 같은 내용을 여러 기능에 복사하지 않고 영향받는 경로를 `PLAN.md`에 나열한다.
- 지식 문서는 YAML 프론트매터의 `summary`와 `read_when`을 제공한다. GitHub 입력 템플릿은 플랫폼 표시를 방해하지 않도록 예외로 한다.

## 문서별 의미

- `AGENTS.md`: 이 기능에서 어떻게 작업하는가
- `SPEC.md`: 현재 무엇이 정답인가
- `DECISIONS.md`: 정답이 왜 바뀌었는가
- `PLAN.md`: 정답을 어떤 단계로 구현하는가
- `QA.md`: 구현 결과를 어떻게 검증했는가
- `HANDOFF.md`: 다음 작업자는 어디서 시작하는가

채팅에서만 합의되고 로컬 문서에 반영되지 않은 내용은 인수인계된 결정으로 취급하지 않는다.

## 변경 유형

작업 조정자는 요청을 다음 중 하나로 분류한다.

| 유형 | 설명 | 기본 처리 |
| --- | --- | --- |
| 설명 보완 | 동작을 바꾸지 않고 문구나 기준을 명확하게 함 | 관련 문서만 갱신하고 검증 |
| 기능 변경 | 한 기능의 동작, 데이터 또는 UI를 바꿈 | 기능 `SPEC.md`, 계획과 QA 갱신 |
| 전역 변경 | 용어, 게임 루프, 공통 규칙 또는 여러 기능을 바꿈 | 루트 기준과 모든 영향 기능을 먼저 갱신 |
| 실험 | 채택 여부를 판단하기 위한 임시 구현 | 정식 SPEC을 바꾸지 않고 실험 범위·종료 조건 기록 |

## 기획 변경 흐름

1. 사용자 지시와 현재 로컬 기준을 확인한다.
2. 변경 전과 변경 후를 한 문장씩 정리한다.
3. 영향받는 기능, 코드, UI, 데이터, QA와 문서를 찾는다.
4. 미확정 결정과 충돌하는지 확인한다.
5. `PLAN.md`에 변경 유형, 단계, 검증과 제외 범위를 기록한다.
6. 사용자에게 계획과 영향 범위를 제시한다.
7. 사용자의 승인·거부·수정 피드백을 기록한다.

### 승인된 경우

1. `SPEC.md`를 코드보다 먼저 갱신한다.
2. 중요한 변경이면 `DECISIONS.md`에 결정 ID를 추가한다.
3. 전역 변경이면 `AGENTS.md`, `GAME_TERMS.md` 또는 `GROUND_RULES.md`를 필요한 범위에서 갱신한다.
4. `PLAN.md`와 `QA.md`의 수용 기준을 새 기획에 맞춘다.
5. 사용자가 승인한 구현 단계만 실행한다.

### 거부되거나 수정된 경우

- 기존 `SPEC.md`는 유지한다.
- `PLAN.md`에 피드백과 계획 수정 이유를 기록한다.
- 단순히 거부된 초안은 `DECISIONS.md`에 장기 기록하지 않는다.
- 수정한 계획을 다시 제시하고 승인 전에는 구현하지 않는다.

### 실험인 경우

- 실험 가설, 범위, 비교 대상, 성공·중단 조건과 되돌리는 방법을 기록한다.
- 실험 결과만으로 정식 `SPEC.md`를 자동 변경하지 않는다.
- 사용자가 채택한 경우에만 정식 결정으로 승격한다.

## 결정 기록

중요한 결정에는 `DEC-YYYYMMDD-NN` 형식의 ID를 사용한다.

```markdown
## DEC-20260723-01 — 결정 제목

- 상태: approved | superseded | reverted
- 요청자:
- 변경 전:
- 변경 후:
- 변경 이유:
- 영향받는 기능:
- 갱신 문서:
- 검증 기준:
- 승인 기록:
```

단순 오탈자, 표현 정리와 구현 세부 조정은 결정 로그를 만들지 않는다.

## 단계별 구현과 피드백

기본 반복은 다음과 같다.

```text
계획
→ 사용자 승인
→ 승인된 한 단계 구현
→ 구현자 자체 검증
→ 변경 검토
→ 사용자 결과 피드백
```

- 계획 전체를 승인받아도 자동으로 모든 단계를 실행하지 않는다.
- 사용자가 여러 단계를 명시적으로 일괄 승인한 경우에만 묶어서 진행한다.
- 검증 실패, 범위 이탈 또는 새 기획 문제가 발견되면 즉시 멈추고 계획 단계로 돌아간다.
- 구현자는 변경 파일, 실행한 명령, 정확한 결과와 남은 위험을 기록한다.

## QA와 버그 처리

1. 모든 구현 단계가 끝나면 `QA.md`에 환경, 정상·실패·예외·회귀 항목을 확정한다.
2. 전문 QA는 구현자의 결론보다 요구사항과 수용 기준을 중심으로 독립 검증한다.
3. 결과는 가능한 경우 수치, 로그, 화면 캡처 또는 녹화와 함께 기록한다.
4. 측정값에는 환경, 단위, 실행 횟수, 입력·시드와 측정 방법을 함께 기록한다.
5. 측정하지 않은 값은 추정하지 않고 `미측정`으로 표시한다.
6. 버그는 Jira 생성 전에도 작업 기록에 `untracked`로 남긴다.
7. 사용자에게 Jira의 `버그` 이슈 생성 여부를 묻고 승인된 경우에만 외부 이슈를 생성한다.
8. 수정 후 관련 항목과 회귀 범위를 다시 검증한다.

## 일자별 기록과 외부 보고

일자별 문서에는 다음 내용을 포함한다.

- 구현 내용
- 구현 결과
- 트러블슈팅: 증상, 재현, 원인, 해결과 재검증
- 사용자 기획·개발 변경으로 개선된 점
- 검증 수치와 조건
- 캡처·영상·로그 등 증거 경로
- 남은 작업과 위험

- 작은 이미지와 로그만 필요한 범위에서 Git에 포함한다.
- 큰 영상이나 바이너리는 저장 위치와 비용을 사용자와 합의하기 전 일반 Git에 추가하지 않는다.
- 일자별 보고가 준비되면 사용자에게 Confluence 게시 여부를 묻는다.
- 승인된 경우에만 사용자가 지정한 페이지를 읽고 기존 내용을 보존하며 갱신한다.

## 인수인계

다음 작업자는 아래 순서로 컨텍스트를 복구한다.

1. 브랜치와 Pull Request 설명
2. 루트 `AGENTS.md`
3. 대상 기능 `AGENTS.md`
4. 관련 `SPEC.md`
5. 최신 관련 `DECISIONS.md`
6. 활성 작업의 `PLAN.md`
7. `HANDOFF.md`
8. `QA.md`
9. 기록된 검증 명령 재실행

`HANDOFF.md`에는 다음을 반드시 기록한다.

- 브랜치와 기준 커밋
- 현재 작업 상태와 마지막 완료 단계
- 완료된 내용과 남은 작업
- 적용된 결정 ID
- 변경 파일
- 실행·검증 명령과 결과
- 알려진 문제와 미검증 항목
- 다음 작업자가 처음 실행할 한 가지 행동

## Git 규칙

- 브랜치 변경·생성, 커밋, Push, Pull Request와 Merge는 사용자 승인 후 수행한다.
- `main`은 기능을 통합하고 테스트하는 공간이다.
- `feature/<name>`은 개별 기능 개발 공간이며 기능이 동작하고 검증된 뒤 `main`에 합친다.
- `deploy`는 현재 배포 빌드, `deploy/<version>`은 버전 보존과 분리 테스트에 사용한다.
- `deploy` 계열 병합은 Pull Request와 Git 관리자의 검토를 거친다.
- 커밋 제목은 `type: English sentence (ISSUE-KEY)` 형식을 사용한다. 이슈가 없으면 키를 만들거나 추측하지 않는다.
- 허용 타입은 `feat`, `fix`, `refactor`, `style`, `chore`, `resource`, `test`다.
- 커밋 본문은 한국어 Markdown으로 작업 의도, 관련 이슈, 결과와 발견 버그를 기록한다.
- 하나의 커밋에는 하나의 설명 가능한 작업 책임만 담는다.

## 인수인계 완료 조건

- [ ] 현재 기획이 `SPEC.md`에 반영됐다.
- [ ] 중요한 변경 이유가 `DECISIONS.md`에 기록됐다.
- [ ] `PLAN.md`의 단계와 승인 상태가 최신이다.
- [ ] 검증 명령과 정확한 결과가 기록됐다.
- [ ] 미검증 사항과 버그가 숨겨져 있지 않다.
- [ ] 다음 첫 행동이 한 문장으로 명확하다.
- [ ] 필요한 변경이 모두 커밋됐다.
- [ ] 브랜치의 책임과 변경 내용이 일치한다.
