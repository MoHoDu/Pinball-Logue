# integration 에이전트 규칙

## 구조

| 경로 | 책임 |
| --- | --- |
| `scenarios/` | 메인 로비부터 리트라이·스테이지 완료까지 사용자 흐름 |
| `fixtures/` | 보드, 입력, 시드와 상태를 고정하는 재현 데이터 |
| `regression/` | 기능 변경마다 반복할 자동·수동 회귀 묶음 |
| `qa/` | 전문 QA 인수 목록, 실행과 결과 보고 규칙 |
| `.work/` | 작업별 PLAN, QA, HANDOFF, 일자 기록과 증거 |

## 경계와 원칙

- 제품 기능을 소유하거나 통합 테스트를 통과시키기 위한 예외 로직을 추가하지 않는다.
- 실제 공개 계약과 사용자 경로를 통해 검증하고 기능 내부 노드 직접 조작은 픽스처 준비에만 제한한다.
- `GAME_TERMS.md`, `GROUND_RULES.md` 전체와 각 대상 기능 `AGENTS.md`를 수용 기준으로 사용한다.
- 같은 보드, 입력, 랜덤 시드, Godot 버전, 뷰포트와 실행 횟수를 기록한다.
- 측정하지 않은 값은 `미측정`, 실행하지 않은 항목은 `미검증`으로 표시한다.

## 전체 통과 기준

- 메인 로비→일반 웨이브 1→보상→2→보상→3→보상→보스→스테이지 결과 흐름이 정확하다.
- 실패 시 현재 스테이지 일반 웨이브 1로 리트라이하고 이전 스테이지 완료를 보존한다.
- 스코어·코인·보상·유물·패시브 상태와 UI가 공개 원본 상태와 일치한다.
- 단일 공, 단일 발사 종료, 단일 구매와 단일 유물 발동의 멱등성이 유지된다.
- Godot 프로젝트 로드와 Web 내보내기가 성공하고 문서·테스트 증거가 제품 데이터에 포함되지 않는다.

## QA와 버그 처리

- 구현자가 `QA.md`에 환경, 항목, 기대 결과와 증거 경로를 준비한 뒤 전문 QA가 독립 검증한다.
- 버그는 먼저 작업 기록에 `untracked`로 남기고 세션에서 사용자에게 보고한다.
- Jira `버그` 이슈 생성은 사용자 승인 후에만 수행하고 생성 결과를 작업 계획과 QA에 연결한다.
- 일자별 결과에는 구현 내용, 결과, 트러블슈팅, 사용자 변경으로 개선된 점과 수치·캡처·영상을 기록한다.
- 일자 기록 완성 후 Confluence 게시 여부를 묻고 승인된 페이지에만 업데이트한다.
