# app 기능 에이전트 규칙

## 상위 규칙

- 루트 `AGENTS.md`를 항상 먼저 읽는다.
- 새 작업, 기획 변경, QA 또는 인수인계는 루트 `WORKFLOW.md`를 따른다.
- 이 문서는 `app/` 아래 전체에 적용한다. 하위 `AGENTS.md`는 해당 책임만 추가한다.

## 책임

- Godot 메인 씬과 애플리케이션 루트 구성
- 게임 시작과 초기화 순서
- 기능별 루트 씬의 생성, 연결과 해제
- 메인 로비, 플레이, 결과 등 최상위 화면 전환
- 앱 종료, 재시작과 복구 가능한 오류의 안전한 화면 복귀
- 최상위 전환 중 중복 입력과 중복 전환 방지

## 제외 범위

| 제외 내용 | 소유 기능 |
| --- | --- |
| 런·스테이지·웨이브·보상·보스의 규칙과 상태 판단 | `game_flow/` |
| 공·플리퍼·발사·충돌과 물리 | `pinball/` |
| 보드 구성·목표와 웨이브 데이터 | `stages/` |
| 개별 화면의 UI 구성과 내부 상호작용 | `ui/` |
| 카메라·이펙트·음향 연출 | `presentation/` |
| 여러 기능에서 공유할 일반 구현 | `shared/` |

`app/`은 기능을 조립하지만 게임 규칙을 계산하거나 보관하지 않는다.

## 하위 계층

| 경로 | 책임 | 추가 문서 |
| --- | --- | --- |
| `bootstrap/` | 메인 씬, 시작 순서와 기능 루트 초기화 | `bootstrap/AGENTS.md` |
| `navigation/` | 최상위 화면 경로, 전환과 전환 보호 | `navigation/AGENTS.md` |
| `.work/` | app 관련 활성 작업의 계획·QA·인수인계 | 작업을 시작할 때만 생성 |

## 문서 라우팅

| --- | --- |
| 루트 `AGENTS.md` | 항상 |
| 루트 `WORKFLOW.md` | 계획, 기획 변경, QA, Git 또는 인수인계 시 |
| 루트 `GAME_TERMS.md` | 로비·런·스테이지·웨이브·결과 화면 이름을 다룰 때 |

## 의존 방향

```text
app
├── game_flow
├── ui
├── presentation
└── shared
```

- 다른 기능 디렉토리는 `app/`을 참조하지 않는다.
- `app/`은 공개된 기능 진입점만 조립하고 기능 내부 구현을 직접 조작하지 않는다.
- 웨이브 클리어, 실패와 보스 완료는 `game_flow/`의 결과를 받아 화면 전환으로 연결한다.
- `app/`을 전역 이벤트 저장소, 서비스 로케이터 또는 공용 코드 보관소로 사용하지 않는다.
- 새 오토로드 추가는 전역 수명과 테스트 격리를 바꾸므로 별도 계획과 사용자 승인이 필요하다.

## 상태 소유권

`app/`이 소유할 수 있는 상태는 기술적인 애플리케이션 상태로 제한한다.

- 시작 중
- 실행 준비 완료
- 화면 전환 중
- 복구 가능한 오류 처리 중
- 종료 중

런, 스테이지, 웨이브, 빌드, 코인과 스코어 상태는 소유하지 않는다.

## 구현 규칙

- 메인 씬은 기능을 조립하는 얇은 루트로 유지한다.
- 화면 전환 요청은 한 진입점으로 모으고 동시에 두 전환을 실행하지 않는다.
- 씬을 해제할 때 시그널, 타이머와 장기 참조가 남지 않게 한다.
- 초기화 순서가 중요하면 코드와 테스트에 순서를 명시하고 암묵적 노드 순서에 의존하지 않는다.
- 실패 시 가능한 경우 메인 로비 또는 명시된 안전 화면으로 복귀한다.
- 기능별 상세 조건문이 `app/`에 늘어나면 해당 소유 기능으로 이동한다.

## 검증

- Godot 프로젝트와 메인 씬이 오류 없이 로드되는가
- 초기화가 한 번만 실행되는가
- 로비 → 플레이 → 결과 → 로비 전환이 가능한가
- 반복 전환 시 중복 노드, 중복 시그널과 남은 타이머가 없는가
- 전환 중 중복 입력이 두 화면을 동시에 만들지 않는가
- 기능 디렉토리가 `app/`을 역참조하지 않는가
- 게임 규칙과 진행 데이터가 `app/`에 저장되지 않는가
- 헤드리스 프로젝트 스캔과 Web 데이터 패키징이 통과하는가
