# Pinball_Logue 에이전트 규칙

## 적용 범위

이 파일은 저장소 전체에 적용한다. 하위 디렉토리에 `AGENTS.md`가 생기면 루트 규칙을 상속하고, 해당 기능에 필요한 추가 규칙만 정의한다.

## 작업 전 필수 확인

1. 사용자의 최신 지시와 작업 범위를 확인한다.
2. 루트부터 대상 디렉토리까지 존재하는 모든 `AGENTS.md`를 읽는다.
3. 아래 문서 라우팅 표에서 관련 문서를 찾는다.
4. 관련 Markdown의 YAML 프론트매터를 먼저 읽고 `summary`와 `read_when`을 기준으로 본문이 필요한지 판단한다.
5. 새 작업, 기획 변경, QA, Git 또는 인수인계라면 `WORKFLOW.md`의 관련 절을 읽는다.
6. 로컬 문서만으로 요구사항을 확정할 수 없으면 외부 자료를 찾지 말고 사용자에게 필요한 정보를 요청한다.
7. 구현 단계와 단계별 검증 항목을 제시하고 사용자 승인을 받은 범위만 작업한다.

## 외부 정보 접근 정책

- 기본 작업 컨텍스트는 저장소 내부의 Markdown과 코드뿐이다.
- 사용자가 현재 요청에서 특정 외부 소스나 링크를 제공하고 읽기·검토·갱신을 명시적으로 요청한 경우에만 해당 소스에 접근한다.
- 이전 대화에서 제공된 링크, 로컬 문서에 과거에 적힌 링크 또는 관련성이 있어 보이는 외부 페이지는 자동으로 열지 않는다.
- 웹 검색, Figma, Google Drive, Jira, Confluence와 GitHub 원격 조회는 기본적으로 금지한다.
- 로컬 정보가 누락되거나 충돌하면 추측이나 검색 대신 사용자 피드백을 요청한다.
- 사용자가 기획 변경 소스를 제공하면 요청된 소스만 읽고, 승인된 변경을 로컬 Markdown에 반영한다. 이후 작업은 갱신된 로컬 문서를 기준으로 수행한다.
- 로컬 문서에는 실행에 불필요한 외부 링크를 보관하지 않는다.

## 기획 기준과 우선순위

- 우선순위는 `사용자의 최신 지시 → 대상 기능 SPEC.md → 대상 디렉토리 AGENTS.md → 루트 AGENTS.md → GROUND_RULES.md → GAME_TERMS.md → 현재 구현`이다.
- 사용자가 기획 변경을 명시하면 관련 로컬 문서를 먼저 갱신하고 그 문서를 구현 기준으로 사용한다.
- 확정된 규칙과 미확정 항목을 섞지 않는다. `D-01`~`D-09`는 사용자 또는 팀의 결정 없이 임의로 구현 규칙을 확정하지 않는다.
- 규칙을 변경해야 하면 영향받는 코드, UI, QA와 문서를 계획에 기록하고 사용자 승인을 받는다.

## 핵심 게임 구조

```text
메인 로비
→ 일반 웨이브 1 → 보상 구매
→ 일반 웨이브 2 → 보상 구매
→ 일반 웨이브 3 → 보상 구매
→ 보스 웨이브
→ 스테이지 클리어
```

- 한 `런`은 현재 승인 기준 스테이지 1→2→3의 전체 진행이다.
- 한 `스테이지`는 일반 웨이브 3개와 보스 웨이브 1개로 구성한다.
- 일반 웨이브는 목표 스코어 달성이 목적이며, 현재 프로토타입에는 일반 몬스터를 사용하지 않는다.
- 스테이지 안의 어느 웨이브에서 실패해도 해당 스테이지의 일반 웨이브 1부터 리트라이한다.
- 보상 화면에는 랜덤 보상 카드 3개를 제시한다. 카드는 무료 1개 선택이 아니라 코인을 지불하는 상품이며, 보유 코인 범위에서 복수 구매할 수 있다.
- 구매한 보상은 최소한 해당 스테이지의 보스 웨이브까지 유지한다.

## 핵심 경험 불변 규칙

- 최우선 재미는 연쇄 충돌, 콤보와 스코어 폭발이다. 첫 플레이 10초 안에 강한 충돌 또는 연속 타격을 경험할 수 있어야 한다.
- 한 발사의 핵심 판단은 `목표 경로 선택`이다. `플리퍼 선택`과 `플리퍼 작동 타이밍`은 선택한 경로를 실행하는 수단이다.
- 판단 순서는 `상황 확인 → 목표 경로 선택 → 플리퍼 선택 → 타이밍 입력`이다.
- 공은 완전히 통제되지 않지만 플레이어의 선택은 결과 확률을 유의미하게 바꿔야 한다.
- 우연성은 예고 없는 손실보다 예상 이상의 충돌 연쇄와 보너스 결과에 우선 사용한다.
- 한 번에 보드에 존재하는 공은 1개다. 현재 프로토타입에는 멀티볼과 공 분열을 사용하지 않는다.
- 유물은 배치 위치에 따라 공의 경로, 스코어 또는 생존 결과를 바꿔야 한다. 위치가 중요하지 않은 효과는 패시브로 분류한다.
- 공과 경로의 가독성을 UI와 이펙트보다 우선한다.
- 보스는 일반 웨이브에서 만든 빌드와 익힌 조작을 검증하며, 프로토타입에서는 대표 기믹 또는 공격 패턴 하나를 중심으로 제작한다.

## 표준 표현

- `월드맵`, `스테이지 선택 화면` 대신 `메인 로비`
- `한 공`, `한 번의 플레이` 대신 `발사`
- `3개 보상 중 1개 선택` 대신 `랜덤 보상 카드 3개 중 코인으로 구매`
- `덱 빌딩` 대신 `보드 빌딩` 또는 `유물 배치 빌드`
- `영구적 죽음` 대신 `스테이지 리트라이`
- 시스템 상태의 `플레이어 사망` 대신 `스테이지 실패`
- 포괄적인 `아이템` 대신 역할에 따라 `웨이브 아이템`, `유물`, `보상 카드`
- 화면은 `보상 선택 화면`, 플레이어의 행동은 `보상 구매`
- `다음 판` 대신 상황에 따라 `다음 웨이브`, `다음 스테이지`, `리트라이`

## 검증 원칙

- 기능 구현 전에 관련 `GR` 항목을 수용 기준으로 선택한다.
- 결과는 가능하면 정확한 수치, 실행 로그, 화면 캡처 또는 녹화로 남긴다.
- 변경 전후를 비교할 때는 같은 보드, 입력, 랜덤 시드와 실행 환경을 사용한다.
- 측정하지 않은 수치는 추정하지 않고 `미측정`으로 기록한다.
- 물리 변경은 터널링, 끼임, 무한 반복, 과속, 플리퍼 관통과 필승 조합을 확인한다.
- QA에서 버그를 발견하면 작업 기록에 먼저 남기고, Jira의 `버그` 이슈 생성은 사용자 승인 후 수행한다.
- 기획 변경, 실험, 작업 기록과 인수인계는 `WORKFLOW.md`를 따른다.
- 프로젝트 스킬의 `SKILL.md`는 스킬 표준에 따라 YAML에 `name`, `description`만 사용하며 일반 지식 문서 프론트매터 검사의 예외다.
