# ui 기능 에이전트 규칙

## 책임과 구조

| 경로 | 책임 |
| --- | --- |
| `lobby/` | 메인 로비와 스테이지 진행 표시 |
| `preparation/` | 유물 확인·배치와 웨이브 준비 |
| `gameplay/` | 스코어·목표·코인·콤보·공 상태 HUD |
| `reward/` | 보상 카드 3개 비교와 구매 입력 |
| `results/` | 웨이브·스테이지 결과와 리트라이 안내 |
| `components/` | 둘 이상의 화면이 쓰는 공용 UI 요소 |

## 경계

```text
app → ui
ui → shared
```

- UI는 공개 상태를 표시하고 사용자 의도를 명령으로 전달할 뿐 게임 규칙, 구매, 물리와 진행 상태를 직접 변경하지 않는다.
- 화면 전환은 `app/`, 상태 판단은 각 도메인 기능, 카메라·이펙트·음향은 `presentation/`이 소유한다.
- 노드 경로로 다른 기능 내부에 접근하지 않고 타입이 명확한 뷰 모델과 명령을 사용한다.

## 공통 규칙

- `GAME_TERMS.md`의 표준 표현을 사용하고 스코어, 코인과 진행 단위를 혼용하지 않는다.
- 공과 경로의 가독성을 장식, 숫자 팝업과 이펙트보다 우선한다.
- 키보드·마우스 입력과 Web 화면 크기 변화에서 핵심 조작과 정보가 유지돼야 한다.
- 비활성·로딩·성공·실패 상태와 입력 차단 이유를 명확히 표시한다.
- 같은 명령을 연속 입력해 구매, 전환 또는 발사가 중복 실행되지 않게 한다.

## 필수 검증

- 각 화면의 필수 정보가 실제 도메인 상태와 일치하는지 확인한다.
- 1280×720 기준과 승인된 최소 Web 뷰포트에서 잘림·겹침을 확인한다.
- 키보드 포커스, 마우스 입력과 입력 연타를 확인한다.
- 스코어와 코인, 다음 웨이브와 다음 스테이지, 낙하와 스테이지 실패가 구분되는지 확인한다.
